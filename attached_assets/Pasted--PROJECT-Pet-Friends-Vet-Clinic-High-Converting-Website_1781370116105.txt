## 4. COPYWRITING DIRECTION
- Headline (H1): lead with the core benefit/transformation — focus on "your pet's second home" / "24/7 care when they need it most", not just "Welcome to Pet Friends"
- Use a clear value proposition above the fold: 24/7 availability, compassionate expert staff, complete care under one roof (vet + grooming + emergency)
- Address pain points → translate into benefits:
  - "Worried about emergencies?" → "Our team is here 24/7, even at midnight"
  - "Anxious pet during grooming?" → "Our groomers handle even the most nervous pets with patience and care"
  - "Confused about vaccination requirements?" → "Clear, honest guidance — we explain everything before you decide"
- Use trust-building language: "thousands of happy pet parents", "experienced veterinarians", "part of the Ragab Elhofy Group network of clinics", "4.5 stars from 1,000+ reviews"
- Every section should end with a soft or direct CTA toward booking an appointment or contacting via WhatsApp/phone
- Microcopy on buttons should be action + benefit oriented (e.g., "Book Your Visit Today", "Chat With Us on WhatsApp", "Call Now — We're Open 24/7", not just "Submit")
- Tone: warm, caring, reassuring, professional — emotional connection ("they're not just pets, they're family") balanced with clinical credibility

## 13. PORTABILITY & DEPLOYMENT REQUIREMENTS (CRITICAL)
This project must be built so it can be re-deployed without errors in two specific scenarios: (1) deploying to Vercel, and (2) re-importing the GitHub repo into a fresh Replit account after running out of free tokens on the current one. To guarantee this:

- Do NOT use Replit-only services or APIs: no Replit DB, no Replit Auth, no Replit-specific environment variable system, no hardcoded paths like `/home/runner/...`
- Use standard environment variables via a `.env` file, and include a `.env.example` listing every required variable with placeholder values
- Use a standard database that works outside Replit (SQLite file-based) — avoid anything tied to Replit's storage
- `package.json` must include standard scripts: `dev`, `build`, `start` (and `vercel-build` if needed), with all dependencies explicitly listed (no implicit/global packages)
- If using a frontend framework, structure the project so it matches Vercel's expected conventions for that framework (e.g., Next.js app/pages directory, or a `vercel.json` with correct `builds`/`routes` for a custom Express + static frontend setup)
- Backend (Express or similar) must be deployable as serverless functions if targeting Vercel, OR clearly documented as needing a separate host (e.g., Render/Railway) if it can't run serverless — pick whichever requires the LEAST manual fixing
- No absolute imports or paths that assume a specific machine/account — use relative paths and environment-based config everywhere
- Include a `README.md` with exact, copy-pasteable steps to: (a) run locally, (b) redeploy on Vercel from scratch, (c) re-import this repo into a new Replit account and have it run with zero manual fixes (only setting env vars)
- Avoid any "magic" auto-configuration that Replit applies invisibly (e.g., Nix-specific setup, Replit's auto-detected run command) — everything needed to run the app must be explicit in config files committed to the repo
